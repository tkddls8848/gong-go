# ECR 규격 자동 추출 계획

본공고 첨부(제안요청서·과업내용서)에서 **ECR(Equipment Composition Requirement, 시스템 장비구성 요구사항)** 항목을 뽑아 구조화된 표로 정리한다. 로컬 도구(`local-dev/`) 안에서만 동작하며 프로덕션 Pages 앱은 건드리지 않는다.

이 문서의 설계 판단은 전부 실측에 근거한다. 첨부 확장자 분포는 수집된 CSV 전체(공고 41.8만건 / 첨부 107만개)를 집계했고, 문서 구조와 변환 품질은 `public/rfp_sample/`의 실제 파일 5건으로 확인했다.

---

## 1. 대상 문서 구조

`rfp_sample`의 문서들을 파싱해 보면 국내 정보화사업 **요구사항 정의서 표준 양식**을 일관되게 따른다. 이 기능이 성립하는 근거다.

| 문서 | 형식 | 쪽수 | ECR 개수 | 번호 체계 |
|---|---|---:|---:|---|
| IoT 기반 탑승교 시설관리 고도화 | pdf | 27 | 13 | `ECR-01` / `ECR-001` 혼용 |
| 4대보험 고객센터 노후장비 교체 | pdf | 46 | 15 | `ECR-001`~`ECR-015` |
| 우진메트로양산 차세대 통합경영정보시스템 | pdf | 162 | **50** | `ECR-001`~`ECR-050` |
| 전자결재 기안기 ActiveX 제거 | pdf | 95 | 3 | `ECR-001`~`ECR-003` |
| SR-MaaS 통합정보시스템 운영환경 구축 | **hwp** | 72 | **42** | **`ECR-COM-01`, `ECR-HW-01`, `ECR-NW-01`, `ECR-SW-01`** |

여기서 두 가지가 바로 드러난다.

**① ECR 개수가 3~50개로 10배 이상 벌어진다.** "문서 하나 = API 호출 하나"라는 전제는 성립하지 않고, 처리량에 따른 분할이 필요하다(4-2).

**② 번호 체계가 표준이 아니다.** SR-MaaS는 서브분류(COM/HW/NW/SW)를 낀 3단 구조라 `/ECR-\d{3}/` 같은 고정 정규식으로는 42개 중 **0개**를 잡는다. 정규식을 정답지로 쓰면 안 된다(4-4).

### 1-1. 두 겹 구조 — 총괄표가 정답지다

문서는 항상 **총괄표 → 상세표** 두 겹으로 되어 있다.

**① 요구사항 총괄표** — ECR 전체 목록이 메타데이터처럼 한자리에 모여 있다. SR-MaaS 문서는 ID 부여규칙과 개수까지 명시한다.

```
구  분                        ID부여규칙   요구사항수
① 시스템 장비 요구사항(ECR)    ECR-000       42
② 성능 요구사항(PER)           PER-00         4
③ 테스트 요구사항(TER)         TER-000        4
...                                          합계 92
```

**② 요구사항 상세 내용** — 각 ECR마다 동일한 골격의 표가 하나씩 있다.

```
요구사항 분류    시스템 장비 구성
요구사항 번호    ECR-COM-01
요구사항 명칭    제안 제품의 공통 요구사항
요구사항 상세설명
  정의          제안 제품의 공통 요구사항 정의
  세부내용      ◦ 제안사는 발주사의 인프라 환경, 목표시스템 구성 및 운영 정책 등을
                  고려하여 HW 및 SW의 품목 및 수량을 제시하고 적합한 제안을 하여야 함
                  - 제안요청서에 별도 명시된 사항이 없더라도 ...
                ◦ 기본규격
                   ┌────┬──────────────────────────────┬────┐
                   │구분│ 요구사항                      │수량│
                   ├────┼──────────────────────────────┼────┤
                   │최소│ CPU 16 core X 2개 이상        │1식 │
                   │사양│ RAM 128GB 이상 / GPU 24GB X 2 │    │
                   └────┴──────────────────────────────┴────┘
  산출 정보     제조사 공급확약서, 설치계획서, 설치결과서
```

**총괄표가 정답지 역할을 한다.** 총괄표에서 ID 집합을 먼저 확보하면 상세 추출 결과와 대조해 **누락을 기계적으로 검출**할 수 있다. *"LLM이 뭔가 빠뜨렸을까"*라는 답 없는 질문이 *"42개 중 41개만 나왔다"*는 확인 가능한 사실로 바뀐다. 실측으로도 확인됐다 — SR-MaaS 총괄표가 선언한 42개와 HWPX에서 실제 추출한 ID 42개가 정확히 일치했다. 아래 검증 설계 전체가 여기에 기댄다.

### 1-2. 파싱 함정 3가지

**① PDF 선형 텍스트 추출은 셀 순서를 뒤섞는다.** 표가 쪽을 넘어가면 순서가 깨진다. 4대보험 문서 PDF 10쪽은 `ECR-010` 표 머리로 시작하는데 중간에 앞 쪽 `ECR-009`의 남은 세부내용(`◦ 문자화된 데이터는 상담원과 고객의...`)이 끼어든다. **어느 문장이 어느 ECR 소속인지 텍스트 순서만으로는 알 수 없다.**

**② 라벨 표기가 문서마다 흔들린다.** `요구사항 분류`(4대보험) vs `요구사항분류`(IoT). `요구사항 번호` 라벨이 추출 결과에서 0회로 잡히는데 ECR 코드는 멀쩡히 존재하는 문서도 있다. **라벨 문자열 매칭 파서는 만들면 안 된다.**

**③ 코드 정규식에 `\b`를 쓰면 안 된다.** 추출 텍스트에서 코드가 한글에 붙는다(`...공통사항ECR-002· 장비 라이선스`). 한글은 `\w`라 `\b`가 성립하지 않아 매칭이 통째로 누락된다 — 첫 스캔에서 15개 중 5개만 잡힌 원인이 이것이었다.

## 2. 처리 경로

### 2-1. 실측 확장자 분포

수집된 본공고 CSV 전체를 집계했다.

| 확장자 | 전체 첨부 (107만개) | **제안요청서·과업내용서만** (19.3만개) |
|---|---:|---:|
| **hwp** (구형, CFB) | 47.6% | **63.5%** |
| **hwpx** (ZIP+XML) | 11.8% | **17.9%** |
| **pdf** | 23.5% | **12.9%** |
| zip | 3.1% | 3.0% |
| xlsx / xls | 13.1% | 2.3% |
| 기타 | 0.9% | 0.4% |

**구형 HWP가 63.5%로 주 경로다.** 코너 케이스가 아니다.

### 2-2. HWP 변환 실측 — HWPX가 압도적

SR-MaaS 샘플(917KB, CFB 시그니처 `D0CF11E0A1B11AE1` 확인 = HWP 5.0)을 한글 COM으로 각 형식에 저장해 비교했다.

| 저장 형식 | 소요 | 크기 | ECR ID 추출 | 불릿 `◦` | 표 구조 |
|---|---:|---:|---:|---|---|
| PDF | 23.5초 | 1,927 KB | **1개** ❌ | — | 쪽 넘김에서 붕괴 |
| HTML | 1.3초 | 3,790 KB | 35개 | `?`로 깨짐 ❌ | 보존(133 table) |
| **HWPX** | **0.7초** | 934 KB | **42개** ✅ | 179회 보존 ✅ | 완전(133 tbl / 1,703 tr / 6,021 tc) |

- **PDF**는 1-2①의 문제가 그대로 재현되고 23초나 걸린다.
- **HTML**은 한글이 cp949로 저장해 `◦`가 `?`로 손실되고 ID도 35개만 잡혔다.
- **HWPX**는 UTF-8에 셀 구조가 XML로 명시되어 손실이 없고, 총괄표가 선언한 42개와 정확히 일치했다.

### 2-3. 결론: HWPX로 정규화

```
HWP  (63.5%) ─ COM으로 HWPX 저장 (0.7초) ─┐
HWPX (17.9%) ─────────────────────────────┼─→ HWPX 파서 하나 → Markdown 표
PDF  (12.9%) ─────────────────────────────── API에 base64로 네이티브 전달
zip  ( 3.0%) ─ 재귀 전개 후 위 규칙 적용
그 외        ─ 건너뛰고 사유 기록
```

**단일 파서(HWPX)가 81.4%를 커버한다.** COM은 "변환기"가 아니라 **HWP를 HWPX로 정규화하는 어댑터**일 뿐이다. 이 역할 분리가 중요한 이유는, 나중에 클라우드로 올려 COM을 못 쓰게 되더라도 HWPX 파서는 그대로 살아남기 때문이다.

**CFB(HWP 5.0) 자체 파서는 만들지 않는다.** 500~600줄이 드는데 COM이 0.7초에 더 정확히 해낸다. 클라우드 이전이 실제로 결정되면 그때 붙인다.

## 3. COM 어댑터

### 3-1. 방식

한글을 설치하면 자신을 `HWPFrame.HwpObject`라는 COM 자동화 서버로 등록한다. 이 PC에는 한컴오피스 2022가 설치되어 등록이 확인됐다(`HKLM:\SOFTWARE\Classes\HWPFrame.HwpObject`, `Hwp.Document.120`).

Node에서 `winax` 같은 네이티브 모듈을 쓰면 **의존성 0 원칙이 깨진다.** 대신 PowerShell 스크립트를 `execFile`로 호출한다.

```powershell
# hwp-to-hwpx.ps1  <입력경로> <출력경로>
$hwp = New-Object -ComObject HWPFrame.HwpObject
$null = $hwp.RegisterModule("FilePathCheckDLL","FilePathCheckerModule")
try { $hwp.XHwpWindows.Item(0).Visible = $false } catch {}
$null = $hwp.Open($args[0], "", "forceopen:true")
$ok = $hwp.SaveAs($args[1], "HWPX", "")
try { $hwp.Clear(1) } catch {}
$hwp.Quit()
if (-not $ok) { exit 1 }
```

`SaveAs`는 **인자 3개**다(`경로`, `형식`, `옵션`). 2개로 호출하면 `Cannot find an overload` 예외가 난다 — 실측에서 처음 걸렸던 지점이다.

호출 측에서는 `execFile`로 인자를 **배열로** 넘긴다. 첨부 파일명에 한글·공백·대괄호가 흔해서 셸을 거치면 깨진다.

### 3-2. 아직 검증 안 된 것

| 항목 | 상태 |
|---|---|
| **보안 모듈 팝업** | `RegisterModule`이 **False**를 반환했다(DLL 미등록). 이 파일은 팝업 없이 열렸지만 대량 처리에서도 안 뜬다는 보장이 없다. **팝업이 뜨면 배치 자체가 불가능하므로 최우선 확인 대상이다** |
| 배포용/암호 문서 | `FileHeader` 플래그 bit1(암호) / bit2(배포용) 문서에서 `SaveAs`가 허용되는지 미확인. 한컴 DRM이라 자체 파서로는 어차피 불가능한 영역 |
| 프로세스 안정성 | GUI 프로세스라 대량 반복 시 누수·멈춤 가능. N건마다 재시작 필요 |

**1단계 다운로드 직후 수십 건으로 이 셋을 먼저 잰다.** 팝업이 막히면 pyhwp(`hwp5html`) 폴백으로 선회한다.

## 4. LLM 추출 설계

### 4-1. 호출 방식

수집기의 의존성 0 원칙을 지켜 `fetch`로 `POST https://api.anthropic.com/v1/messages`를 직접 호출한다. `collector.js:402`의 `loadEnv()`를 재사용해 `.env`에서 `ANTHROPIC_API_KEY`를 읽는다(`.env`는 이미 gitignore 대상).

```
headers: {
  "x-api-key": ANTHROPIC_API_KEY,
  "anthropic-version": "2023-06-01",
  "content-type": "application/json"
}
```

모델은 `claude-opus-4-8`. 규격 누락이 곧 실패인 작업이라 정확도를 우선한다.

```js
{
  model: "claude-opus-4-8",
  max_tokens: 64000,                       // ECR 50개 × 세부내용 원문 = 수만 토큰
  stream: true,                            // max_tokens가 크면 스트리밍 필수
  thinking: { type: "adaptive" },          // 4.8은 명시해야 켜진다
  output_config: {
    effort: "high",
    format: { type: "json_schema", schema: ECR_SCHEMA }
  },
  system: [{ type: "text", text: SYSTEM_PROMPT }],
  messages: [{ role: "user", content: [ ...문서블록, { type: "text", text: 지시문 } ] }]
}
```

- PDF는 `{ type: "document", source: { type: "base64", media_type: "application/pdf", data } }`로, **텍스트 블록보다 앞에** 둔다.
- `temperature` / `top_p` / `top_k`는 **넣지 않는다** — Opus 4.8에서 400이 난다.

### 4-2. 2패스 + 대조

1-1에서 확인한 총괄표를 그대로 활용한다.

**Pass A — 총괄표 추출.** 문서 전체를 넣고 요구사항 목록만 받는다. 출력이 작아(2K 토큰 내외) 저렴하다.
```
{ 사업개요: {...}, ID부여규칙: "ECR-COM-NN 형식", 요구사항목록: [{ 구분, ID, 명칭 }] }
```
번호 체계가 문서마다 다르므로(1절) **ID를 문자열 그대로** 받는다. 정규화하지 않는다.

**Pass B — ECR 상세 추출.** Pass A에서 얻은 ID 목록을 프롬프트에 **명시적으로 넣어** "이 ID들을 빠짐없이 채우라"고 지시한다. 빈칸 채우기 문제로 바뀌어 누락이 크게 줄고, 모델이 표를 스스로 찾아 헤맬 필요가 없다.

**대조.** `Pass A ID 집합 − Pass B ID 집합` = 누락 목록. 비어 있지 않으면 해당 ID만 재요청한다. 3회 후에도 남으면 `누락` 필드에 기록하고 뷰어에서 경고로 노출한다.

**ECR 20개 초과 시** Pass B를 ID 구간별로 쪼갠다(SR-MaaS 42개 → 서브분류 COM/HW/NW/SW 단위 4회, 우진메트로 50개 → 15개씩 4회). 이때 문서 블록에 `cache_control: { type: "ephemeral" }`을 걸어 두 번째 호출부터 입력을 캐시에서 읽는다(0.1배). 연속 호출이라 기본 5분 TTL로 충분하다. **20개 이하 단일 호출에서는 캐시를 걸지 않는다** — 쓰기 프리미엄 1.25배만 내고 읽을 일이 없어 손해다.

### 4-3. 출력 스키마 (`spec-schema.js`)

```
{
  사업개요: { 사업명, 수요기관, 사업기간, 예산, 계약방식 },
  ID부여규칙: 문자열,
  요구사항목록: [{ 구분, ID, 명칭 }],        // 총괄표 원본 = 대조용 정답지
  ecr: [{
    id,                      // "ECR-COM-01" — 문서 표기 그대로
    분류,                    // "COM" / "HW" / "NW" / "SW" — 서브분류 있으면
    명칭,
    정의,
    세부내용_원문,           // 세부내용 셀 원문 그대로 (가공 금지)
    기본규격: [{ 구분, 항목, 요구사항, 수량 }],   // 중첩표 구조화
    파생규격: { cpu, ram, gpu, disk, nic, 기타 },  // 세부내용에서 뽑은 값
    산출물: [문자열],
    출처: 문자열,            // 원문 위치 (섹션/쪽) — 사람이 대조할 때 필요
    불확실: [문자열]
  }],
  누락: [문자열],
  기타요구사항: [{ 구분, ID, 명칭 }]         // PER/SER/TER 등은 ID·명칭만
}
```

설계 의도 셋:

- **`세부내용_원문`은 원문 그대로 보존한다.** 요약하면 검증하려고 결국 원본을 다시 열게 된다. 원문이 그대로 있으면 환각 여지가 거의 없고 자동 검증도 가능하다.
- **`기본규격`과 `파생규격`을 분리한다.** 전자는 문서에 실제로 표로 있는 것, 후자는 모델이 해석해 뽑은 것이다. 섞으면 어디까지가 원문인지 알 수 없다.
- **ECR 외 접두어는 ID·명칭만 담는다.** PER/SER/TER/QUR/COR/PMR/PSR 등이 함께 쓰이지만 대상은 ECR이다. 목록은 남겨 나중에 범위를 넓힐 때 스키마를 안 건드려도 되게 한다.

ECR이 사내 표준 양식(고정 컬럼 셋)을 따라야 한다면 **이 파일만 교체하면 되고** 나머지 단계는 그대로 쓴다.

### 4-4. 자동 검증 (`verify.js`)

1절 ②에서 확인했듯 번호 체계가 문서마다 다르므로 **정규식은 정답지가 아니다.** 검증은 문서가 스스로 선언한 총괄표를 기준으로 한다.

1. **ID 집합 대조** — Pass A 집합 == Pass B 집합인가. 이것이 1차 방어선이다.
2. **개수 대조** — 총괄표가 개수를 명시한 경우(SR-MaaS `42`) 그 숫자와 일치하는가.
3. **원문 대조** — `세부내용_원문`의 앞 40자가 변환 Markdown에 실제로 존재하는가. 공백 정규화 후 검색한다. 없으면 환각이다.
4. **느슨한 형식 확인** — `/([A-Z]{2,4}[-–][A-Z0-9-]+)/` 로 뽑은 후보와 비교해 **경고만** 낸다. 실패로 처리하지 않는다.

1~3을 통과한 결과에만 `verified: true`를 붙이고, 뷰어는 이 플래그로 배지 색을 구분한다.

## 6. 단계별 실행

### 1단계 — 첨부파일 다운로드 (`attachments.js`)

여기까지만 해도 "관심 공고 첨부 일괄 확보"라는 독립적 효용이 있어 먼저 한다.

- `analyze.config.json`으로 대상 선정. `fileNamePattern`이 핵심이다 — 본공고 첨부에는 입찰유의서·계약일반조건이 섞여 있어 전부 넣으면 토큰만 낭비된다. 이 패턴을 적용하면 대상이 107만개에서 19.3만개로 준다(2-1).
  ```json
  {
    "mode": "bid",
    "begin": "2026-07-01",
    "institutions": ["국민건강보험공단", "건강보험심사평가원"],
    "fileNamePattern": "제안요청|과업내용|과업지시|규격|사양",
    "maxNoticesPerRun": 50,
    "concurrency": 4
  }
  ```
- 이미 받은 파일은 건너뛴다(경로 존재 + 크기 > 0). 실패는 `download-errors.json`.
- 파일명은 `basename` 후 `[<>:"/\\|?*]`와 제어문자를 제거한다. API가 준 이름을 그대로 경로에 쓰지 않는다.
- 응답 `Content-Type`이 `text/html`이면 로그인/오류 페이지이므로 실패 처리(나라장터가 종종 이렇게 응답한다).

**검증:** 최근 1주 본공고로 실행 → 임의 HWP를 한글에서 열어 정상 확인.

### 2단계 — 정규화 + 변환 (`convert.js`, `hwpx-table.js`)

**2a. COM 어댑터 검증 (게이트).** 1단계로 받은 `.hwp` 50건에 `hwp-to-hwpx.ps1`을 돌려 3-2의 세 항목을 잰다. **팝업 발생률·성공률·평균 소요를 기록하고, 팝업이 뜨면 여기서 멈춘다.** 이 계획의 최대 리스크다.

**2b. HWPX 파서.**
- `zip-read.js`: EOCD → Central Directory → Local File Header 순회 + `zlib.inflateRawSync`. hwpx/docx/xlsx/zip이 전부 ZIP이라 한 번 만들어 네 곳에서 쓴다.
- `hwpx-table.js`: `Contents/section*.xml`에서 문단과 `<hp:tbl>`/`<hp:tr>`/`<hp:tc>`를 순회해 Markdown 표로 뽑는다. 병합 셀(`colSpan`/`rowSpan`)은 값을 복제해 표를 직사각형으로 유지한다. 실측 기준 SR-MaaS 문서는 표 133개 / 행 1,703개 / 셀 6,021개다.
- 산출물: `.md.gz` + `manifest.json`(`kind`, `source`, `converted`, 사유, 바이트 수). 3단계는 manifest만 읽으면 되므로 두 단계가 느슨하게 결합된다.

**검증:** `csv-record.test.js`와 같은 방식으로 `node --test` 추가. `rfp_sample`의 SR-MaaS를 fixture로 넣고 **ECR ID 42개가 전부 잡히고 각 `세부내용`이 올바른 ID에 귀속되는지** 확인한다(1-2①이 재발하지 않는지 보는 테스트다). 불릿 `◦` 179회 보존도 함께 확인한다 — 이게 깨지면 인코딩 경로가 잘못된 것이다.

### 3단계 — 추출기 (`analyze.js` + `spec-schema.js` + `verify.js`)

4절 그대로 구현한다. 운영상 필요한 것들:

- **비용 가드:** 실행 전 대상 문서 크기를 합산해 예상 비용을 출력하고 확인을 받는다. `--yes`로 생략, `--dry-run`, `--limit N`도 함께 둔다.
- **재개:** `analysis-state.json`에 완료 공고번호를 기록해 재실행 시 건너뛴다. `--force`로 무시.
- **메타 기록:** `model`, `usage`(캐시 적중 포함), `analyzedAt`, `sourceFiles`, `schemaVersion`. `schemaVersion`이 있어야 스키마를 바꿨을 때 재분석 대상을 골라낼 수 있다.
- **오류:** 429/5xx는 지수 백오프 재시도(`collector.js:131` `requestJson` 패턴 그대로). `stop_reason`이 `refusal` / `max_tokens`인 경우를 분기해 기록한다 — 후자는 `max_tokens` 부족이나 구간 분할 실패 신호다.

**검증:** `rfp_sample` 5건을 정답셋으로 쓴다. ECR 개수는 이미 실측해뒀다(13 / 15 / 50 / 3 / 42). 4-4의 1~3번을 5건 모두 통과해야 하며, 특히 **SR-MaaS(서브분류 ID 체계)와 우진메트로(50개, 구간 분할)** 두 건이 관문이다.

### 4단계 — 뷰어 통합 (`public/app.js`, `index.html`)

- `analyze.js`가 실행 끝에 `analysis-index.json`(공고번호 → 경로, ECR 개수, `verified`)을 갱신한다.
- `app.js`가 `index.json`과 함께 읽어 결과 행에 배지 표시(`ECR 42` / 검증 실패 시 경고색).
- 첨부 모달에 탭 2개(`첨부파일` / `ECR 규격`). ECR 탭 첫 진입 시 JSON을 fetch(지연 로딩)해 표로 그린다.
  - 기본 열: ID · 분류 · 명칭 · 수량 · 산출물
  - 행 클릭 시 `세부내용_원문` + `기본규격` 중첩표 펼침 (원문이 길어 접어둬야 표가 안 망가진다)
  - `누락`과 `불확실`은 표 위에 경고로 고정
- CSV 다운로드에 "ECR 펼쳐서 내보내기" 추가 — 실무에서는 이 형태로 쓸 일이 많다.
- 모달 하단에 **"1차 참고용이며 최종 확인은 원문"** 문구를 고정 표기(`index.html:139` 본공고 subtitle과 같은 톤).

**검증:** `python -m http.server 8788` → 배지 → 탭 → 펼침 → CSV까지 확인.

### 5단계 — 온디맨드 분석 (선택)

정적 페이지는 Node를 못 돌리므로 지금은 CLI로 미리 돌려두는 방식이다. 화면에서 버튼으로 분석하고 싶어지면 `local-dev/server.js`(Node 내장 `http`)로 `public/`과 `collector/data/`를 서빙하며 `POST /api/analyze`만 추가한다. 실행법이 `python -m http.server` → `node server.js`로 바뀌므로 README 갱신이 따라온다. **API 키는 서버 쪽에만 둔다.**

지금 할 필요는 없다. 1~4단계로 쓰다가 CLI 반복이 실제로 번거로워지면 그때 한다.

## 7. 비용

Opus 4.8 입력 $5 / 출력 $25 (1M 토큰당). 아래는 **실측 문자 수**에 한국어 약 1.2자/토큰을 적용한 추정이다.

| 문서 | 입력 경로 | 변환 후 문자 | 입력 토큰 | Pass A+B 합계 |
|---|---|---:|---:|---:|
| 4대보험 (46쪽, ECR 15) | HWPX→MD | 약 7만 | 약 6만 | 약 **$0.5** |
| SR-MaaS (72쪽, ECR 42) | HWP→HWPX→MD | **10.6만**(실측) | 약 9만 | 약 **$1.1** |
| 우진메트로 (162쪽, ECR 50) | HWPX→MD, 4구간 + 캐시 | 약 15만 | 약 13만 | 약 **$1.6** |
| 전자결재 (95쪽, ECR 3) | HWPX→MD | 약 8만 | 약 7만 | 약 **$0.5** |

같은 SR-MaaS를 PDF로 보내면 72쪽 × 약 2K = **14.4만 토큰**으로 입력이 60% 늘고 정확도는 오히려 떨어진다(2-2). **HWPX 경로가 더 싸면서 더 정확하다** — 흔치 않은 조합이라 이 선택은 재고할 여지가 없다.

50건 배치면 대략 **$25~55**. 무시할 금액이 아니라서 실행 전 확인 프롬프트를 필수로 둔다.

절감 수단은 효과 순으로:

1. **`fileNamePattern`으로 대상을 좁힌다.** 107만 → 19.3만개(2-1). 가장 싸고 효과가 크다.
2. **HWPX 경로를 우선한다.** PDF 대비 입력 토큰 60% 절감(위).
3. **Message Batches API** (`POST /v1/messages/batches`) — 50% 할인. 소급 분석은 실시간일 이유가 없다. `--batch` 플래그로 추가.
4. **프롬프트 캐싱** — 구간 분할이 일어날 때만 의미가 있다(4-2).
