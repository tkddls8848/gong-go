# API item 전체 CSV 저장 설계

## 목표

로컬 수집기는 API 호출을 줄이기 위한 캐시로서, 나라장터 API의
`response.body.items.item` 배열에 들어 있는 각 공고 객체의 필드와 값을
누락 없이 날짜별 CSV에 보존한다.

프론트엔드의 검색과 결과 테이블은 현재 공통 화면용 열을 계속 사용하며,
표시 열은 변경하지 않는다.

## 데이터 구조

각 CSV 행은 다음 두 종류의 열을 가진다.

1. 앱 공통 메타데이터: `id`, `mode`, `noticeType`, `announcementNumber`,
   `institution`, `businessType`, `title`, `publishedAt`, `closeAt`, `files`,
   `updatedAt`.
2. 해당 API `item` 객체의 모든 최상위 키.

메타데이터는 기존 UI와 중복 제거, 날짜별 파일 분류를 위해 유지한다.
API 항목의 필드는 화면용으로 선별하거나 명세에 하드코딩하지 않는다.

## 수집 및 저장 흐름

1. 각 API 호출의 `response.body.items.item` 배열을 가져온다.
2. 배열의 각 객체에서 기존 공통 메타데이터를 계산한다.
3. 객체 자체의 모든 최상위 `key: value`를 보존한다.
4. 현재 저장 대상 행에서 발견한 API 키들의 합집합을 CSV 헤더로 만든다.
5. 키가 없는 행은 해당 열에 빈 값을 기록한다.
6. 값이 배열 또는 객체이면 JSON 문자열로 하나의 CSV 셀에 기록하고,
   원시 값은 문자열 표현으로 기록한다.

날짜별 CSV를 다시 쓸 때에도 그 파일에 이미 존재하던 API 열을 유지하고,
새로 발견한 열을 추가한다. 따라서 이후 동기화가 과거 데이터의 원본 열을
삭제하지 않는다.

## 프론트엔드 호환성

`local-dev/public/app.js`는 공통 메타데이터 열만 읽는다. CSV 파서는
추가 열을 그대로 객체 속성으로 읽어도 무시하므로, 검색, 결과 테이블,
첨부파일 모달, CSV 다운로드의 표시 구조는 바꾸지 않는다.

## 오류 처리와 검증

- `item`이 아닌 값은 API 응답 오류로 처리하며 기존 오류 기록 흐름을 따른다.
- 동일한 API 키와 공통 메타데이터 키가 충돌하면, 공통 메타데이터를 유지하고
  원본 값은 충돌 없는 별도 이름으로 저장한다.
- 테스트는 서로 다른 키 집합의 `item`들을 저장할 때 모든 열과 값이 남는지,
  JSON 값이 손실 없이 직렬화되는지, 기존 UI용 열을 읽는 기존 CSV가 계속
  파싱되는지를 검증한다.
